## Context

当前项目已经具备 CDC 接收、Redis 消费、Web 展示等多个方向的实现，但它们并没有围绕一份稳定契约收口：

- `CdcHttpServer` 的事件查询接口返回分页对象。
- `CdcWebUIClient` 仍按旧的数组结构消费事件。
- `CdcWebUI` 维护了一套单独的 API 与页面实现。
- 多个文件存在明显乱码，说明文本编码未统一。

P0 的目标不是增加新功能，而是先把“主线、契约、编码”三件事定下来，为 P1/P2 提供稳定基础。

## Goals / Non-Goals

**Goals**

- 明确项目唯一推荐主线。
- 为 CDC 事件列表、事件详情、统计信息定义统一 DTO。
- 统一前后端对分页、字段命名与操作类型的理解。
- 统一中文文档风格与 UTF-8 编码。

**Non-Goals**

- 本阶段不处理 Redis 消费职责拆分。
- 本阶段不引入新的持久化存储。
- 本阶段不做完整测试体系建设。

## Decisions

### 1. 主线选择

采用 `CdcHttpServer + CdcWebUIClient` 作为唯一推荐主线。

原因：

- 后端边界更清晰。
- 前端可以独立演进。
- 更适合后续补测试、补配置与做部署说明。

### 2. 统一 API 契约

CDC 查询接口统一为以下风格：

- `GET /api/events`
  - 返回分页对象，而不是裸数组。
- `GET /api/events/{id}`
  - 返回统一事件详情 DTO。
- `GET /api/stats`
  - 返回统一统计 DTO。

建议统一事件 DTO 字段：

- `id`
- `sourceSchema`
- `sourceTable`
- `op`
- `before`
- `after`
- `tsMs`
- `receivedAt`
- `rawJson`

前端展示层如需 `tableLabel`、`timeText` 等字段，应在前端派生，而不是要求后端输出两套结构。

### 3. 操作类型保持底层语义

后端仍使用 Debezium 原始操作类型值 `c/u/d/r` 作为接口值。

原因：

- 与 CDC 上游语义一致。
- 避免后端和 Redis 消费端再做一层不必要转换。
- 前端可根据需要映射为中文标签或 `CREATE/UPDATE/DELETE/READ`。

### 4. 编码与文档约定

仓库中新增与修复的文本文件统一使用 UTF-8 编码。

文档与过程说明约定：

- 中文为主。
- 英文仅用于代码标识、配置键名、命令与外部系统名称。

## Risks / Trade-offs

- **风险：归档 `CdcWebUI` 会影响已有使用者**
  - 缓解：先标记为“非推荐入口”，不立即删除。
- **风险：接口契约调整会导致前端短期不兼容**
  - 缓解：本变更要求后端与前端同步改动。
- **权衡：保留 `op` 原始值会让前端多一层映射**
  - 接受：这能减少跨系统语义漂移。
