# 项目整理清单

## 说明

本项目后续的文档、需求说明、变更过程、任务拆解与沟通记录，默认以中文为主。

如需补充英文内容，建议采用以下方式：

- 中文为正文，英文作为补充说明。
- 面向外部系统、第三方配置项、代码标识符时，保留原始英文名称。
- 对外接口字段命名是否使用英文，以技术一致性为准，但配套文档说明优先使用中文。

## P0

- 统一后端 API 契约与前端 TypeScript 类型。
  - 当前问题：
    - `CdcHttpServer` 的 `/api/events` 返回 `PagedResult<CdcEvent>`。
    - `CdcWebUIClient` 直接把 `/api/events` 当作 `CdcEvent[]` 使用。
    - 后端字段结构与前端字段结构不一致。
  - 交付目标：
    - 明确定义事件列表、事件详情、统计信息的统一响应结构。
    - 后端输出模型与前端 `src/types/cdc.ts` 同步调整。
    - 明确 `/api/events` 的分页契约，不再依赖隐式约定。

- 统一 UI 架构，只保留一条主线。
  - 当前问题：
    - `CdcWebUI` 自带 API 和静态页面。
    - `CdcWebUIClient` 又依赖 `CdcHttpServer` 提供 API。
    - 两套 UI 方案并存，且数据模型并不一致。
  - 交付目标：
    - 二选一：
      - 保留 `CdcHttpServer + CdcWebUIClient`
      - 或保留 `CdcWebUI` 作为单体应用
    - 未选中的方案标记为归档或移除。
    - README、启动方式、开发流程只保留一种推荐路径。

- 修复仓库文本编码，统一为 UTF-8。
  - 当前问题：
    - 多个 `.cs`、`.vue`、`.md` 文件存在乱码。
  - 交付目标：
    - 将受影响的源码和文档统一转换为 UTF-8。
    - 验证终端日志、README、前端文案显示正常。
    - 补充编辑器或团队协作约定，避免再次出现编码漂移。

## P1

- 将 Web 服务与后台 CDC 消费职责拆开。
  - 当前问题：
    - `CdcHttpServer/Program.cs` 里通过 `Task.Run` 启动 Redis 消费逻辑。
    - `RedisProcess.cs` 是常驻循环，生命周期管理较弱。
  - 交付目标：
    - Redis 消费由 `CdcRedisConsumer` 独立承担。
    - `CdcHttpServer` 仅负责 HTTP 接收与查询接口。
    - 如果必须进程内运行后台任务，改为标准 Hosted Service。

- 统一配置管理。
  - 当前问题：
    - 部分参数仍写死在代码或零散地从环境变量读取。
  - 交付目标：
    - 将端口、Redis 配置、Stream Key、事件存储容量、日志路径放入 `appsettings`，并支持环境变量覆盖。
    - 按需补充 `appsettings.Development.json` 或示例配置文件。
    - 去除与个人机器路径强绑定的配置和文档。

- 用结构化日志替代临时文件追加写入。
  - 当前问题：
    - 请求处理中直接使用 `File.AppendAllText` 写日志。
  - 交付目标：
    - 全面使用 `ILogger`。
    - 原始事件持久化改为可选能力，并允许通过配置开关控制。
    - 关键日志保留级别、上下文与问题排查所需字段。

- 统一错误响应与健康检查。
  - 当前问题：
    - `Results.Problem` 当前传入内容不够规范。
    - `/health` 仅返回简单字符串。
  - 交付目标：
    - API 错误统一返回结构化 Problem Details。
    - 健康检查体现 Redis、核心服务等依赖状态。

## P2

- 优化事件存储设计。
  - 当前问题：
    - `InMemoryEventStore` 适合演示，但不适合长期保留与复杂查询。
  - 交付目标：
    - 本地演示继续保留内存实现。
    - 如果有事件留存需求，补充持久化实现。
    - 明确保留策略、过滤需求与基础索引方向。

- 为 CDC 解析与 API 契约补充自动化测试。
  - 建议覆盖：
    - Debezium 事件解析。
    - `/api/events`、`/api/events/{id}`、`/api/stats`。
    - Redis 消费器的处理器匹配与 ACK 行为。

- 明确项目命名与职责边界。
  - 当前问题：
    - `CdcHttpServer`、`CdcWebUI`、`CdcRedisConsumer` 名称本身可理解，但职责边界仍略模糊。
  - 交付目标：
    - 为每个项目补一行中文职责说明。
    - 在 README 和解决方案结构中体现一致边界。

- 强化本地启动文档。
  - 交付目标：
    - 只保留一套推荐启动流程。
    - 提供一份最小可运行示例数据或 SQL。
    - 补充 Debezium、Redis Stream Key、API 检查相关排障说明。

## 推荐执行顺序

1. 先确定目标架构，只保留一条主线。
2. 再统一 API 契约。
3. 然后修复编码与文档。
4. 接着把运行配置移出代码。
5. 再拆分 Web 与 Consumer 的职责。
6. 最后补测试，再继续扩展功能。

## 完成标准

- 项目只有一种清晰的本地启动方式。
- 后端与前端共享同一份权威 API 契约。
- 仓库内不存在造成困惑的重复 UI 路径。
- 源码、README、日志、界面中的中文显示正常，无乱码。
- 生产代码中不再出现机器专属硬编码路径。
- CDC 解析与查询接口具备基础自动化验证。
